Security Techniques

Operational security starts with known-good configurations and controlled endpoints. This section combines baselines, hardening, wireless/mobile administration, and application-security practices.

Core lesson

  • Establish, deploy, monitor, and maintain security baselines
  • Hardening servers, workstations, mobile, embedded, IoT, cloud, and network devices
  • Least functionality, secure defaults, patching, and configuration enforcement
  • MDM, BYOD, COPE, CYOD, geofencing, remote wipe, and mobile application controls
  • Wireless site surveys, channel planning, rogue AP detection, WPA3, RADIUS, 802.1X, and EAP
  • Application input validation, secure cookies, code signing, sandboxing, static/dynamic analysis, and secure SDLC practices

Exam focus

  • A baseline defines the approved state; hardening changes a system toward that state.
  • EAP-TLS uses certificates and supports mutual authentication.
  • Input validation should occur server-side even when client-side validation exists.
  • Code signing supports publisher authenticity and integrity, not confidentiality.

Objective 4.2 · 10 practice questions

Asset Management

You cannot secure assets you do not know exist. Track hardware, software, data, owners, locations, configurations, licenses, and lifecycle status from procurement through disposal.

Core lesson

  • Procurement and approval
  • Inventory, tagging, ownership, and assignment
  • Asset criticality and classification
  • Software inventory, licenses, versions, and unauthorized applications
  • Lifecycle states, end-of-life, and end-of-support
  • Data/media retention, sanitization, cryptographic erase, degaussing, shredding, and destruction
  • Disposal documentation and chain of custody

Exam focus

  • Deletion and formatting may leave recoverable data.
  • Choose sanitization based on media type, sensitivity, and whether reuse is required.
  • End-of-support increases exposure because fixes may no longer be available.

Objective 4.3 · 18 practice questions

Vulnerability Management

Vulnerability management is a recurring process: discover, validate, analyze, prioritize, remediate, verify, and report. Technical severity is only one input; exposure, exploitability, asset value, and business impact determine priority.

Core lesson

  • Credentialed and non-credentialed scans; agent and agentless approaches
  • Application, network, cloud, container, and infrastructure-as-code scanning
  • Static analysis, dynamic analysis, package analysis, and fuzzing
  • Threat intelligence from OSINT, sharing communities, vendors, dark web, and internal observations
  • Penetration-test scope, rules of engagement, exploitation, cleanup, and responsible disclosure
  • CVE, CVSS, vulnerability databases, false positives, and exposure factors
  • Patching, segmentation, compensating controls, exceptions, risk acceptance, and validation scans

Exam focus

  • A scan identifies likely weaknesses; a penetration test attempts controlled exploitation.
  • Prioritize an internet-facing actively exploited flaw over an isolated higher-base-score flaw when context supports it.
  • Rescanning verifies remediation; closing the ticket alone does not.

Objective 4.4 · 15 practice questions

Security Monitoring

Security monitoring turns telemetry into detection and response. Understand what data is collected, where it is centralized, how it is normalized and correlated, and which platform can alert, automate, or investigate.

Core lesson

  • Log aggregation, normalization, correlation, alerting, reporting, and retention
  • SIEM, SOAR, IDS/IPS, EDR/XDR, UEBA, DLP, and network monitoring
  • SCAP, secure configuration assessment, and vulnerability reporting
  • Dashboards, thresholds, baselines, and anomaly detection
  • Continuous monitoring and scheduled reviews
  • Time synchronization and timestamp reliability
  • Alert tuning, false positives, false negatives, and escalation

Exam focus

  • A SIEM correlates and searches events; SOAR automates and orchestrates response.
  • A false positive is benign activity flagged as malicious; a false negative is missed malicious activity.
  • NTP supports timeline correlation but does not make logs tamper-proof.

Objective 4.5 · 18 practice questions

Enterprise Security

Enterprise security applies layered controls across network boundaries, web and email traffic, operating systems, data flows, and endpoints. Select controls based on what must be allowed, inspected, blocked, or recorded.

Core lesson

  • Firewall rules, objects, ports, protocols, screened subnets, and change review
  • Web proxies, content filtering, URL reputation, DNS filtering, and secure web gateways
  • Active Directory, Group Policy, SELinux, and operating-system policy enforcement
  • Secure protocol and port selection, transport encryption, and VPN tunnels
  • SPF, DKIM, DMARC, mail gateways, attachment sandboxing, and anti-phishing controls
  • DLP, file-integrity monitoring, USB control, and data monitoring
  • Endpoint posture, health checks, EDR, host firewalls, allow lists, and isolation

Exam focus

  • SPF authorizes sending systems; DKIM signs message content/headers; DMARC defines alignment, policy, and reporting.
  • DNS filtering blocks resolution; a proxy can inspect and control web requests.
  • Least-permissive firewall rules allow required traffic and deny the rest.
  • EDR can investigate and isolate endpoints; antivirus alone is narrower.

Objective 4.6 · 18 practice questions

Identity and Access Management

IAM manages who receives access, to what, under which conditions, for how long, and with what evidence. Separate identity proofing, authentication, authorization, provisioning, federation, and privileged access.

Core lesson

  • Identity proofing, account provisioning, modification, review, and deprovisioning
  • Single sign-on, federation, SAML, OAuth, OpenID Connect, LDAP, Kerberos, RADIUS, and TACACS+
  • RBAC, ABAC, rule-based, discretionary, and mandatory access control
  • Least privilege, need to know, separation of duties, time/location restrictions, and just-in-time access
  • Authentication factors: knowledge, possession, inherence, location, and behavior
  • MFA, phishing-resistant methods, certificates, hardware tokens, biometrics, and risk-based authentication
  • Password managers, passwordless authentication, complexity, length, history, and lockout policies
  • PAM, vaulting, session recording, and privileged-account controls

Exam focus

  • OAuth delegates authorization; OpenID Connect adds identity/authentication; SAML commonly carries enterprise web SSO assertions.
  • Two passwords are not MFA because both are knowledge factors.
  • Biometric false acceptance affects security; false rejection affects usability.
  • Disable access promptly during offboarding and review dormant accounts.

Objective 4.7 · 10 practice questions

Automation and Orchestration

Automation improves speed and consistency but can repeat mistakes at scale. Know good automation use cases, required safeguards, and the difference between automating a task and orchestrating a multi-system workflow.

Core lesson

  • User and resource provisioning
  • Guardrails and policy enforcement
  • Security-group and firewall changes
  • Ticket creation, enrichment, containment, and notification
  • Continuous integration and deployment checks
  • APIs, webhooks, scripts, playbooks, and infrastructure as code
  • Benefits: speed, consistency, scalability, repeatability, and reduced manual error
  • Risks: complexity, dependency failures, insecure credentials, broad permissions, and unintended changes

Exam focus

  • Automation performs repeatable actions; orchestration coordinates actions across systems.
  • Use testing, approvals, logging, least privilege, and rollback for high-impact automation.
  • Do not hard-code credentials in scripts; use managed secrets.

Objective 4.8 · 18 practice questions

Incident Response

Incident response is a controlled cycle from preparation through lessons learned. Questions often ask for the next or first action, so distinguish identification, containment, eradication, recovery, communications, and evidence requirements.

Core lesson

  • Preparation, detection/analysis, containment, eradication, recovery, and lessons learned
  • Isolation, segmentation, account disabling, and short- versus long-term containment
  • Playbooks, roles, escalation, legal, communications, and stakeholder coordination
  • Tabletop exercises, simulations, and after-action reviews
  • Root-cause analysis and control improvements
  • Forensic acquisition, order of volatility, legal hold, e-discovery, chain of custody, and evidence hashing
  • Preservation versus immediate operational restoration tradeoffs

Exam focus

  • Contain active spread before eradication unless safety or legal direction changes the priority.
  • Do not analyze original media when a verified forensic image is available.
  • Chain of custody records who handled evidence, when, where, and why.
  • Lessons learned occurs after recovery and should update controls and playbooks.

Objective 4.9 · 13 practice questions

Security Data Sources

Different data sources answer different investigative questions. Learn what each source records, how fields relate, and why timestamps, context, retention, and integrity determine evidentiary value.

Core lesson

  • Firewall, IDS/IPS, proxy, DNS, VPN, and network-flow logs
  • Application, API, web server, database, and authentication logs
  • Endpoint, EDR, operating-system, antivirus, and file-integrity telemetry
  • Cloud audit, control-plane, storage-access, and identity logs
  • Packet captures and metadata
  • Common fields: source/destination, port, protocol, action, user, process, parent process, timestamp, and event ID
  • Centralization, normalization, time zones, integrity, retention, and access control

Exam focus

  • A packet capture provides packet-level detail; flow data summarizes conversations.
  • Firewall “deny” records show attempted traffic, not successful compromise.
  • Correlate multiple sources before drawing conclusions from one event.
  • Normalize time zones before building an incident timeline.
